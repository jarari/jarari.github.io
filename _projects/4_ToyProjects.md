### 1. 카메라 및 총기 관성
{% include elements/video.html id="6k_rZS9hRXA" %}

#### 아이디어
- 레퍼런스: [Bodycam](https://www.youtube.com/watch?v=OJtv52GuSWM)
- 비슷한 묵직한 느낌을 주면서도 게임 플레이에는 크게 방해되지 않을 정도의 카메라 무빙과 총기 모델 회전을 구현하는데에 집중
- 폴아웃 4에는 additive 애니메이션의 블렌딩을 통해 카메라가 회전했을 때 1인칭 모델에 어느정도의 관성이 적용되고 있으나, 총기마다 애니메이션 스타일이 통일되어 있지 않아 그렇게 현실적으로 다가오지는 않았기 때문에 코드를 통해 구현

#### 구현 시 애로사항

- 카메라는 코드와 애니메이션에서 동시에 관여하고 있기 때문에 직접 제어하려면 함수의 호출 순서, 타이밍이 중요
- **사이드 이펙트를 최소화**하기 위해 런타임에서 **카메라에 부모 노드를 삽입**하여 해당 노드를 제어

### 2. 좌우 기울이기

![타르코프 스타일](assets/uneducatedshooter_tarkov.webp "타르코프 스타일") | ![시즈 스타일](assets/uneducatedshooter_r6s.webp "레인보우식스: 시즈 스타일")

#### 아이디어
- Rainbow Six: Siege, Escape From Tarkov 같은 택티컬 슈터 장르의 leaning and peeking 메카닉

#### 구현 시 애로사항
- 폴아웃 4 기본 게임에도 피킹이 존재하나, 1인칭에서 화면 앞에 장애물을 둔 채 정조준 시 플레이어의 이전 좌표를 기록해두고 엄폐물에서 가장 가까운 모서리 방향으로 플레이어를 강제 이동시키는 기괴한 방식으로 구현됨
    - 해당 기능을 **xbyak으로 패치하여 강제 비활성화 (jmp)**
- Additive 애니메이션으로 처리하는 것이 일반적이지만, 당시 havok behavior 디컴파일 툴이 특정 클래스의 메모리 구조를 잘못 정의하여 재컴파일에 실패하고 있었음
- 스켈레톤 노드를 직접 움직이려 하면 havok 애니메이션 레이어를 후킹하여야 하지만, 이 경우 **해당 스켈레톤의 주인이 플레이어인지 아닌지 판별하기가 힘듦**
    - 런타임에서 1인칭, 3인칭 스켈레톤의 **특정 노드들에 부모 노드를 삽입**하여 기울이기 시 해당 **노드들을 회전**시키는 방식으로 구현
    - 카메라의 경우 **R6S와 EFT가 서로 다른 방식**으로 구현되어 있음. R6S는 카메라의 수평이 유지된 채 위치만 이동하고, EFT는 카메라가 같이 돌아감. **R6S 모드 사용 시 루트본을 기준으로 포지션을 재계산**하는 로직을 추가하여 수평 상태에서 카메라만 이동되도록 구현
- 카메라만 이동시키니 **적들이 피킹 중인 플레이어를 탐지하지 못하는 문제**가 발견
    - 피킹 로직 자체를 리팩토링하여 플레이어가 실제로 이동하도록 구현
    - **플레이어 이동 후 루트본은 필요한 만큼의 전위를 적용**하여 시각적으로는 가만히 있는 것 처럼 보이게 함
<br>
<br>

### 3. 플레이어의 자세에 따른 콜리전 보정

#### 문제점

#### 구현 시 애로사항
- 플레이어 스케일, 스켈레톤 충돌 스케일이 따로 놀아서 정확한 사이즈를 적용하기 힘듦
    - **Head, Pelvis, Root 본을 캐싱**해두고 세 본의 위치 차이로 **충돌체를 역산하는 나이브한 방식**을 채택
    - 바닐라 플레이와 모션 추가 모드까지 적용한 상태에서 문제가 없었기 때문에 **구현 스펙 상 좋은 접근 방식**이라고 생각

# Auto Beam
{% include elements/video.html id="WN_E_tuR1CA" %}

- 총기 부착물 레이저가 실제 에임 위치를 조준하도록 개선하는 모드
- 누적 다운로드 수 **250,000** (2026.03 기준)

#### 문제점
- 무기 부착물로 사용되는 레이저 포인터가 실제 에임 위치를 조준하지 않음

#### 구현 시 애로사항
- 레이저를 정확히 에임에 꽂히게 만들면 **총은 움직이는데 레이저는 조준점에 붙어있는 문제**가 발생
    - **최근 15프레임 동안의 레이저 각도를 샘플링하여 평균값**을 사용하도록 함
- 총을 넣고 달리기, 재장전 등 특정 모션에서는 에임 방향을 향하면 안되지만, 또 모드로 추가된 총기의 경우 전방을 바라보며 재장전하는 경우가 있기 때문에 이 때는 보정을 해주어야 함
    - **총구 방향과 에임 방향간의 내적, 플레이어의 상태 등을 이용하여 보정 여부를 결정**하도록 함

# Shield Framework

- 폴아웃 4에서 현실적인 건파이트를 위해 **콜 오브 듀티/배틀필드 스타일의 전술 방패**를 구현
- 크리에이션 엔진의 **Havok 물리 엔진**을 이용하여 총알 투사체와 근접 공격을 감지
- 유저들이 방패의 충돌체, 방어 가능 데미지, 방어 시 효과 등을 **json 파일로 커스터마이징**하여 어떤 형태, 모양, 재질의 무기/방어구든 현실적인 피드백을 제공하는 방패로 직접 구현할 수 있는 프레임워크
- 누적 다운로드 수 **120,000** (2026.03 기준)

### 애니메이션 - Telegraphing Attack

![공격 속도 보정](assets/immersiveimpact_speed.gif)

#### 문제점
- 모션의 커브가 완만하다
- 칼을 휘두를 때 강약의 차이 없이 일정한 속도로 이동

#### 아이디어
- 몬스터헌터나 소울류의 게임은 공격 모션의 전조 동작이 크다
- Telegraphing Attack
    - 전조 동작의 길이가 길 수록 **"더 강한 공격이 온다."** 라는 느낌
- 스카이림은 공격 속도로 모션 속도를 조절 가능
- 애니메이션에는 공격의 각 구간에 애니메이션 이벤트가 존재

#### 해결 방안
- **애니메이션 이벤트 기반 공격 속도 보정 시스템 개발**
- 각 애니메이션 이벤트 처리 타이밍에 애니메이션 단계, 무기 타입에 따라 공격 속도를 조절

#### 구현 시 애로사항
- 스크립트를 사용하여 구현하였더니 스카이림의 Papyrus 스크립트는 한 프레임 내에 처리할 수 있는 연산의 한계량 (budget)이 정해져 있음
- 따라서 C++를 이용한 플러그인으로 애니메이션 이벤트 처리 가상 함수를 후킹하여 엔진에서 네이티브로 돌아가게 만드는 것이 빠른 반응을 위한 핵심
<br>
<br>

### 타격감 - Hit Stop, Blur, Camera Shake

{% include elements/video.html id="0EzfUSYoKZE" %}

#### 문제점
- 스카이림에는 타격 시 피드백이 부족하다

#### 아이디어
- 타격감에 필요한 요소들을 정리
    - 타격 순간 일시정지, 화면 집중선 (블러), 카메라 쉐이크
- 모든 요소가 엔진 기능으로 구현 가능 (IMAD - 포스트 프로세싱, 카메라 쉐이크 이벤트)
- 컨트롤러 진동은 덤

#### 구현
- 히트 이벤트를 후킹하여 타격 시점에 타격감을 높여줄 효과들을 재생
- 무기 타입, 공격 타입에 따라 다른 강도로 적용
- IMAD 폼을 플러그인 로드 시 캐싱해두고 엔진에서 적용하는 함수를 호출
- 카메라 쉐이크는 엔진의 쉐이크 함수를 이용

#### 구현 시 애로사항
- IMAD 적용 함수, 카메라 쉐이크 함수 등이 플러그인 프레임워크에 존재하지 않아 **IDA 및 치트엔진을 활용하여 직접 리버싱**하였음
- 해당 함수를 찾기 위해 **동적 분석을 활용**하였으며, 스크립트로 특정 행동 시 화면 이펙트와 카메라 쉐이크를 작동하게 하여서 **콜스택을 통해 호출되는 함수를 분석**
<br>

### 대쉬와 넉백

![대쉬](assets/immersiveimpact_dash.gif)
![넉백](assets/immersiveimpact_knockback.gif)

#### 문제점
- 스카이림에는 캐릭터가 맞았을 때 피격 연출이 특정 상황에서만 발동된다
- 플레이어가 공격할 때 wasd로 움직이면 하체와 상체가 따로 놀며 스케이팅을 타는 듯한 모션이 나오는데, 해당 방식이 공격에서 느껴지는 가속도를 완전히 없애버린다

#### 아이디어 및 구현
- 공격 시 플레이어를 제자리에 고정
- 애니메이션을 만들 기술은 없으므로 캐릭터를 밀어버려서 맞았다는 느낌을 준다
- 플레이어는 공격할 때 앞으로 대쉬한다

#### 구현 시 애로사항
- 스카이림의 캐릭터 컨트롤러는 땅에 붙어있는 캐릭터의 경우 외부의 가속도를 완전히 무시한다
- 공중에서 조작을 가하는 경우 고정된 가속도가 덮어씌운다
- **캐릭터용 별개의 물리 레이어** 및 **엔진 틱에 도는 관리 시스템**을 구현하여 가속도가 필요한 경우 따로 연산하여 캐릭터 컨트롤러에 적용

#### 아이디어
- Apex Legends의 그래플링 훅 액션
- 벽에 박힐 수 있는 Arrow 타입 투사체, Unity의 line renderer와 비슷한 역할을 하는 bendable spline을 이용하면 훅과 로프를 만들 수 있음
- 누적 다운로드 수 **8,300** (2026.03 기준)

#### 구현 시 애로사항
- Immersive Impact 구현 시와 같이 베데스다 특유의 캐릭터 컨트롤러 자체 문제
    - 동일한 가속도 시스템을 구현하여 플레이어가 훅으로 빨려들어가는 가속도를 추가
- 훅을 사용한 채로 세이브/로드 시 세션 간 데이터가 이전되지 않음
    - 세이브 파일 시리얼라이즈 인터페이스를 이용하여 세이브파일에 그래플링 훅 상태 정보가 세이브/로드 되도록 함
